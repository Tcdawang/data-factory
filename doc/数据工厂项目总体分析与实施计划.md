# 数据工厂项目总体分析与实施计划

## 一、项目定位

数据工厂项目定位为：

```text
企业内部的数据处理流水线平台 + 数据服务开放平台
```

核心解决的问题是：

```text
数据从哪里来
数据怎么加工
任务怎么编排
任务怎么执行
执行日志怎么追踪
结果怎么对外提供服务
```

本项目不做传统的数据资产管理平台，也不做数据标准管理平台。当前重点应放在：

```text
数据源管理
脚本管理
任务编排
任务执行
开放接口
执行日志
```

## 四、核心业务流程

### 1. 数据源配置流程

```text
进入数据源管理
-> 新增数据源
-> 选择数据源类型
-> 填写连接信息
-> 测试连接
-> 保存
-> 后续任务节点引用该数据源
```

### 2. 脚本配置流程

```text
进入脚本管理
-> 新增脚本
-> 选择脚本类型
-> 编写脚本内容
-> 配置输入输出参数
-> 调试运行
-> 保存脚本
-> 后续任务节点引用该脚本
```

### 3. 任务编排流程

```text
进入任务管理
-> 新建任务
-> 进入 DAG 画布
-> 拖拽节点
-> 配置节点参数
-> 配置节点输入输出映射
-> 配置节点连线
-> 保存草稿
-> 测试运行
-> 发布任务
```

### 4. 任务执行流程

```text
触发任务
-> 获取任务当前发布版本
-> 解析 DAG / DSL
-> 创建任务执行记录
-> 按 DAG 顺序执行节点
-> 记录每个节点执行日志
-> 汇总任务执行结果
-> 返回输出
```

### 5. 开放接口调用流程

```text
创建开放接口
-> 绑定已发布任务
-> 配置接口路径和请求参数
-> 发布接口
-> 外部系统调用接口
-> 网关接收请求
-> 触发对应任务执行
-> 返回任务执行结果
-> 记录接口调用日志
```

### 6. 发布回滚流程

```text
任务草稿版本
-> 测试通过
-> 发布版本
-> 当前版本生效
-> 记录发布记录
-> 如新版本异常
-> 选择历史版本
-> 回滚
-> 更新当前版本
-> 记录回滚记录
```

## 五、原型页面分析计划

针对 Axure 原型页面，分析计划如下：

```text
1. 使用 Playwright 打开原型页面
2. 如果需要登录，暂停并提示手动登录
3. 逐个分析左侧菜单和页面结构
4. 记录每个页面：
   - 页面名称
   - 表格字段
   - 查询条件
   - 操作按钮
   - 弹窗内容
   - 表单字段
   - 页面跳转
   - 交互流程
5. 将原型页面与 doc 文档中的功能模块做对照
6. 找出：
   - 文档有但原型没有的功能
   - 原型有但文档没写清楚的功能
   - 需要后端支撑但当前需求未明确的点
```

原型分析重点页面：

```text

数据源管理
脚本管理

日志管理
```

明确排除：


## 九、任务 DAG / DSL 设计

### 1. DAG JSON

用于前端画布回显：

```json
{
  "nodes": [
    {
      "id": "start_1",
      "type": "START",
      "name": "开始",
      "position": {
        "x": 100,
        "y": 100
      },
      "config": {}
    }
  ],
  "edges": [
    {
      "source": "start_1",
      "target": "api_1"
    }
  ]
}
```

### 2. DSL JSON

用于后端执行：

```json
{
  "taskCode": "customer_sync",
  "version": "1.0.0",
  "nodes": [
    {
      "key": "api_1",
      "type": "API",
      "input": {},
      "output": {},
      "next": ["script_1"]
    }
  ]
}
```

### 3. DAG 和 DSL 分离原因

```text
dag_json：服务前端画布，关注展示和交互
dsl_json：服务后端执行，关注节点顺序、参数、依赖
```

这样前端画布改动不会直接影响执行器。

## 十、版本管理设计

任务版本状态：

```text
DRAFT
TEST_PASSED
PUBLISHED
ARCHIVED
```

发布状态：

```text
UNPUBLISHED
PUBLISHED
DISABLED
```

测试状态：

```text
UNTESTED
PASSED
FAILED
```

基本规则：

```text
1. 任务可以有多个版本
2. 只有测试通过的版本可以发布
3. 同一个任务同一环境只能有一个当前发布版本
4. 发布时更新 df_task.current_version_id
5. 回滚本质上是将 current_version_id 指向历史版本
6. 每次发布/禁用/回滚都写 publish_record
```

## 十四、风险点

### 1. DAG 执行复杂度

风险：

```text
节点依赖
并行执行
失败中断
条件分支
参数传递
```

建议：

```text
第一版先做线性/简单 DAG
后续再支持复杂拓扑
```

### 2. 脚本执行安全

风险：

```text
用户脚本可能访问系统资源
可能死循环
可能执行危险命令
```

建议：

```text
第一版先只保存脚本和模拟执行
后续引入沙箱或隔离执行环境
```

### 3. 开放接口安全

风险：

```text
接口被外部随意调用
参数污染
高频调用拖垮执行器
```

建议：

```text
后续增加鉴权、签名、限流
```

### 4. 版本发布一致性

风险：

```text
发布过程中数据不一致
current_version_id 更新失败
发布记录缺失
```

建议：

```text
发布、回滚必须加事务
```

### 5. 日志量增长

风险：

```text
执行日志、节点日志增长快
```

建议：

```text
后续按时间归档或清理
```
